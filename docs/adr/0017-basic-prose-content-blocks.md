# ADR-0017: Basic prose for hero and gîte descriptions via a narrower Lexical node set

**Status:** Accepted

Reuses the Lexical rich-text infrastructure built for [ADR-0015](./0015-poi-rich-text-detail.md) (storage shape, public renderer, admin editor, HTML translation bridge) and the auto-translate seam from [ADR-0016](./0016-auto-translate-on-save.md). Refines [ADR-0003](./0003-jsonb-i18n-columns.md) the same way ADR-0015 does.

## Context

`hero_description` and `description` (the "Over de gîte" text) are two `content_block` rows shaped as plain `LocalizedText` (`{ nl, en?, fr?, de? }` strings), edited via a plain `<textarea>` and rendered as a single `<p>{text}</p>` in `hero.tsx` / `gite.tsx`. The owner wants basic formatting — bold, italic, links, multiple paragraphs — in these longer-form texts.

ADR-0015 already solved "rich text that survives per-locale machine translation" for POI detail: serialized Lexical `EditorState` JSON, a fixed node set declared once (`src/lib/lexical/nodes.ts`), a Lexical-free public renderer (`rich-text-renderer.tsx`), an admin `RichTextEditor`, and an HTML translation bridge (`lib/lexical/bridge.ts`) so Google Translate's `text/html` mode preserves inline formatting. `resolveLocalizedDetail` (`src/lib/localized-detail.ts`) is already field-agnostic — it takes any `SerializedEditorState` source/stored pair and has no POI-specific code.

Both target fields live in the generic `content_block` table (`key` → typed `value` jsonb), not POI's per-row columns, so the reuse question is about _value shape_ and _feature set_, not about rebuilding the pipeline.

## Decision

Hero and gîte descriptions become rich text using the **same storage/render/translate pipeline as POI detail, restricted to a narrower node set**, stored as a new `content_block.value` variant.

- **Node set — "basic prose", shared by both fields.** Bold, italic, paragraphs, and links only. No headings, no lists. Both hero and gîte use the identical restricted set (no per-field variation) — headings/lists would fight the surrounding page structure (hero sits under a large image in a fixed-width strip; the gîte section already has its own heading), and a single shared set keeps one thing to maintain instead of three. `src/lib/lexical/nodes.ts` gains a second node array (e.g. `BASIC_EDITOR_NODES = [LinkNode]`, since bold/italic are built-in text formats needing no node class) alongside the existing `EDITOR_NODES`. `RichTextEditor` gains a `variant: "basic" | "full"` prop controlling which node array is registered and which toolbar buttons render (heading/list buttons hidden for `"basic"`); it defaults to today's POI behavior so no existing call site changes. `RichTextRenderer` needs **no change** — it already renders whatever node types it's handed and simply never receives heading/list nodes for these two fields. The HTML bridge (`bridge.ts`) also needs **no change** — its headless editor stays registered with the full `EDITOR_NODES` set for parsing, which is a harmless superset since basic-field content can only ever be a subset.
- **Schema — new union variant on the existing generic table, not new columns.** `ContentBlockValue` gains `LocalizedEditorStateValue = { type: "localizedEditorState" } & LocalizedEditorState`, alongside today's `LocalizedText | ImageUrl`. `content_block` stays a true key → typed-value table; POI's separate `detail`/`detailSource` columns exist because a POI row already carries several unrelated fields (title, body, image, distance) — content_block rows don't, so a third value variant is the smaller, more consistent change. No DDL migration: `value`/`value_source` are already untyped `jsonb`.
- **Migration — one-off write-back script, not a permanent read-time shim.** `scripts/migrate-content-blocks-to-rich-text.mts` reads the two existing rows and, for each present locale, wraps the plain string into a single-paragraph `SerializedEditorState` using a bare `@lexical/headless` editor (`$createParagraphNode` + `$createTextNode`, no DOM), then writes the result back as `LocalizedEditorStateValue`, leaving `valueSource` untouched. It does **not** go through `bridge.ts`'s `htmlToEditorState` — `bridge.ts` imports the bare specifier `"server-only"`, which only resolves inside Next's bundler alias; a plain `tsx` script fails to resolve it at all. Building the paragraph/text nodes directly needs no DOM and sidesteps the issue entirely, while still producing JSON the real Lexical runtime generated (not hand-rolled), so it's guaranteed schema-correct. After it runs once, every reader (`hero.tsx`, `gite.tsx`, the admin content page) only ever handles the new shape — no indefinite `type === "localizedText"` fallback branch. `e2e/content.spec.ts`'s direct-insert seeds must be updated to the new shape in the same change.
- **Translation — switch to the HTML bridge, re-translate on next save.** `updateHeroDescriptionAction` / `updateDescriptionAction` swap `upsertLocalizedText` (plain-text `resolveAuthoredField`) for `resolveLocalizedDetail` (already generic, ADR-0015/0016), matching POI's `text/html` translation path so bold/italic/links survive. Existing `en`/`fr`/`de` machine translations are superseded the next time the owner saves either field — consistent with `*_source` already meaning "machine slots are overwritten whenever the source changes"; the migration script does not proactively re-translate, it only wraps existing text so nothing is lost before the next save.
- **No other consumers.** Confirmed by search: only `hero.tsx`, `gite.tsx`, and the admin `content/` route (`page.tsx`, `content-client.tsx`, `actions.ts`) read or write these two keys. Page `<meta description>` values come from separate i18n message-file strings (`t("description")`), not these content_block rows, so SEO metadata is unaffected.

## Consequences

- Admin editing for these two fields moves off the TanStack-form `nl: string` field onto the same `useState<SerializedEditorState>` + manual `FormData` pattern POI's editor already uses (ADR-0015's note that a recursive `EditorState` type blows up TanStack's `DeepKeys` inference applies here too). Since a hero/gîte form has exactly one field (the rich text itself, no sibling fields like POI's title/distance), TanStack form added no value once that field left it — `content/shared.ts` (the form-options/schema module) is deleted rather than kept half-used.
- `RichTextEditor`'s toolbar becomes slightly more complex (a `variant` prop instead of a fixed button set), but `RichTextRenderer` and the HTML bridge stay untouched — the restriction is enforced entirely at the authoring edge (which nodes the editor can create), not at render or translation time.
- Because `resolveLocalizedDetail` and the bridge functions were already field-agnostic (no POI-specific code in either), this ADR requires no changes to `localized-detail.ts` or `bridge.ts` at all — the only new code is the node-set/toolbar variant, the schema union variant, the two call-site swaps, and the migration script.
- If a third free-text field later wants rich formatting, it reuses `BASIC_EDITOR_NODES`/`variant="basic"` directly; a fourth distinct feature set (e.g. only bold/italic, no links) would be the trigger to generalize `variant` into a more granular feature-flag shape.
